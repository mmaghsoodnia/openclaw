### Gateway startup hang — diagnosis and fix

After deploying the merge to VPS, the gateway hung at startup: 100%+ CPU, no logs, no HTTP port binding, health checks failing. Same behavior reproduced on local staging.

**Root cause:** Upstream commit `8e0ab35b0e` ("refactor(plugins): decouple bundled plugin runtime loading") replaced pre-generated static import files with runtime filesystem scanning + jiti TypeScript transpilation. The Dockerfile sets `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` which points to source `.ts` files. The new code uses jiti to synchronously transpile all 13+ channel plugins (each importing 15-25 SDK subpaths) at startup, blocking the Node.js main thread indefinitely.

**Security audit:** Ran a full audit of all 29 upstream commits. No malicious code, obfuscation, backdoors, or credential exfiltration found. The hang is a Docker-specific regression, not a security threat.

**Fix:** Set `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/dist/extensions` in docker-compose overrides to point to the pre-compiled `.js` files instead of source `.ts`.

- Applied to `docker-compose.staging.yml` (checked into git)
- Applied to VPS `docker-compose.override.yml` (runtime, not in git)
- Gateway starts in seconds with the fix, binds port, all channels connect

## Key lesson

When upgrading upstream, watch for changes to plugin/extension loading paths. The `OPENCLAW_BUNDLED_PLUGINS_DIR` env var in Docker must point to compiled `.js` output (`dist/extensions/`), not source `.ts` files (`extensions/`). This will need to be maintained in future upgrades until upstream fixes the Dockerfile default.

# Session Log — 2026-03-12

## Summary

Fixed DOCKER-USER iptables rule that was blocking all container outbound traffic (Telegram, Google APIs, etc.). The original rules from 2026-03-10 were missing a conntrack RELATED,ESTABLISHED rule, causing return packets from external APIs to be dropped.

## What Was Done

### Diagnosis

- User reported Telegram bots unreachable
- VPS SSH confirmed gateway was up and healthy, port 18789 listening
- Gateway logs showed `deleteWebhook failed: Network request for 'deleteWebhook' failed!` in a loop
- `wget https://api.telegram.org` from inside container: FAILED
- Root cause: DOCKER-USER chain's final DROP rule was catching return traffic from outbound connections (packets from external IPs arriving via Docker bridge don't match Tailscale, lo, or 172.16.0.0/12 rules)

### Fix Applied

Added `conntrack --ctstate RELATED,ESTABLISHED` RETURN rule before the DROP:

```
iptables -F DOCKER-USER
iptables -A DOCKER-USER -i tailscale0 -j RETURN
iptables -A DOCKER-USER -i lo -j RETURN
iptables -A DOCKER-USER -s 172.16.0.0/12 -j RETURN
iptables -A DOCKER-USER -m conntrack --ctstate RELATED,ESTABLISHED -j RETURN
iptables -A DOCKER-USER -j DROP
```

## Lesson Learned

When writing DOCKER-USER iptables rules for Docker, always include `conntrack --ctstate RELATED,ESTABLISHED` before the final DROP. Docker containers initiate outbound connections through the bridge network, and the return traffic doesn't match interface-based rules (like `tailscale0`) or Docker-internal source ranges. Without conntrack, you break all outbound connectivity from containers.
